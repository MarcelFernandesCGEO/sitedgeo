masterfile

Introdução​

O Masterfile é um padrão em JSON para descrever a modelagem de um banco de dados espaciais. Para cada especificação é definido um arquivo masterfile, que pode ser convertido de forma automática para a linguagem de interesse, por exemplo SQL para criação de um banco PostGIS, ou a SQL de criação de um Geopackage.

Definição do Masterfile​

O Masterfile é dividio em 4 seções:

  • Informações gerais: informações sobre o modelo, nome de schemas, sufixo ou prefixo de geometrias, etc.
  • Domínios: definição dos códigos e valores dos domínios.
  • Classes: definição das classes com suas geometrias e atributos.
  • Classes auxiliares: definição das classes auxiliares que não fazem parte da especificação.

Informações Gerais​

As informações gerais são os atributos básicos que descrevem a modelagem. São eles:

  • modelo: Nome do modelo descrito pelo masterfile.
  • versao: Versão do modelo.
  • schema_dados: Para exportação para um banco de dados é o schema onde estarão as classes.
  • schema_dominios: Para exportação para um banco de dados é o schema onde estarão as domínios.
  • coord_sys: EPSG do sistema de coordenadas dos dados.
  • fill_factor: Para exportação para um banco de dados PostGIS é o Fill Factor padrão das tabelas.
  • nome_geom: Nome da coluna geométrica nas classes.
  • nome_id: Nome da coluna de id nas classes.

Abaixo um exemplo de preenchimento desses atributos:

"modelo": "EDGV 3.0 Pro",
"versao": "1.00",
"schema_dados": "edgv",
"schema_dominios": "dominios",
"coord_sys": 31982,
"fill_factor": 80,
"nome_geom": "geom",
"nome_id": "id",

Para definição dos nomes das classes baseado no tipo geométrico se utilizam sufixos ou prefixos.

Exemplo de caso de sufixo, por exemplo utilizado na EDGV:

"geom_suffix": {
  "MultiPoint": "_p",
  "MultiLinestring": "_l",
  "MultiPolygon": "_a"
},

Exemplo de caso de prefixo, por exemplo utilizado na TRD 4.4 (MGCP):

"geom_prefix": {
  "MultiPoint": "p", 
  "MultiLinestring": "l", 
  "MultiPolygon": "a"
}

Também se pode definir o valor de "A SER PREENCHIDO", valor padrão de domínio para um atributo que não foi preenchido até o momento pelo operador. A utilidade de tal valor é não utilizar "Desconhecido" para valores ainda não preenchidos.

"a_ser_preenchido": {
  "code": 9999,
  "value": "A SER PREENCHIDO"
}

Finalmente, se pode definir o atributos padrão, que serão aplicados em todas as classes. A descrição do atributo segue o mesmo padrão que será explicado posteriormente na seção sobre Classes.

 "atributos_padrao": [
  {
    "nome": "observacao",
    "tipo": "varchar(255)",
    "cardinalidade": "0..1"
  }
]

Domínios​

Domínios armazenam pares códigos e valores de forma a descrever a listar os valores possíveis de serem utilizados em um atributo de uma classe. Não é necessário especificar o valor 'a ser preenchido'.

{
  "nome": "tipo_curva_nivel",
  "valores": [
    {
      "code": 1,
      "value": "Mestra"
    },
    {
      "code": 2,
      "value": "Normal"
    },
    {
      "code": 3,
      "value": "Auxiliar"
    }
  ]
}

Os domínios também pode ser organizados por grupos, que são utilizados como filtro na implementação dos formulários no QGIS.

 {
  "nome": "tipo_veg",
  "valores": [
    {
      "code": 150,
      "value": "Cult - Arroz de inundação",
      "valor_filtro": "Vegetação Cultivada"
    },
    {
      "code": 194,
      "value": "Cult - Perene",
      "valor_filtro": "Vegetação Cultivada"
    },
    {
      "code": 196,
      "value": "Cult - Anual",
      "valor_filtro": "Vegetação Cultivada"
    },
      {
      "code": 201,
      "value": "Mangue",
      "descricao": "-",
      "valor_filtro": "Mangue"
    },
  ],
  "filtro": true
},

Classes​

A definição de classes utiliza quatro propriedades básicas:

  • nome: Nome da classe sendo descrita.
  • categoria: Categoria para fins de agrupamento de classes similares. O nome da classe no formato de destino será a concatenação entre a categoria, o nome, e o tipo geométrico (baseado no prefixo e sufixo das informações gerais). Por exemplo: aquisicao_limite_massa_dagua_l.
  • primitivas: Lista de primitivas geométricas que aquela classe pode assumir. Os seguintes valores são válidos: MultiPoint, MultiLinestring, MultiPolygon.
  • atributos: Lista de atributos da classe.

Exemplo:

{
  "nome": "limite_massa_dagua",
  "categoria": "aquisicao",
  "primitivas": ["MultiLinestring"],
  "atributos": []
},

As propriedades geom_suffix e geom_prefix podem ser definidas para a classe, e esta irá sobrepor a definição geral.

Na lista de atributos cada atributo deve ser descrito individualmente. Existem 3 propriedades básicas:

  • nome: Nome do atributo.
  • tipo: tipo do atributo, sendo valores válidos varchar, smallint, real, boolean e integer. Para varchar deve ser descrito seu tamanho, por exemplo varchar(80).
  • cardinalidade: Cardinalidade do atributo, sendo valores válidos: 0..1, 1..1, 0..* e 1..*
{
  "nome": "nome",
  "tipo": "varchar(255)",
  "cardinalidade": "0..1"
},

Para associar um atributo com um domínio se deve utilizar a propriedade mapa_valor que recebe como valor o nome do domínio desejado.

{
  "nome": "tipo",
  "tipo": "smallint",
  "cardinalidade": "1..1",
  "mapa_valor": "tipo_veg"
},

Pode se restringir valores do mapa de valor utilizando a propriedade valores, e listando os valores desejados.

{
  "nome": "material_construcao",
  "tipo": "smallint",
  "cardinalidade": "1..1",
  "mapa_valor": "material_construcao",
  "valores": [
    {
      "code": 0
    },
    {
      "code": 1
    },
    {
      "code": 2
    },
    {
      "code": 4
    },
    {
      "code": 23
    }
  ]
}

Também a seguinte simplificação é aceitável.

{
  "nome": "material_construcao",
  "tipo": "smallint",
  "cardinalidade": "1..1",
  "mapa_valor": "material_construcao",
  "valores": [ 0, 1, 2, 4, 23 ]
}

Pode se restringir certos valores a primitivas geométricas adicionando a propriedade primitivas, e listando as primitivas desejadas. Se um código não possui essa propriedade implica que ele é válido para todas as primitivas.

{
  "nome": "tipo",
  "tipo": "smallint",
  "cardinalidade": "1..1",
  "mapa_valor": "tipo_elemento_infraestrutura",
  "valores": [
    {
      "code": 701,
      "primitivas": ["MultiPolygon", "MultiLinestring"]
    },
    {
      "code": 801,
      "primitivas": ["MultiPolygon", "MultiLinestring"]
    },
    {
      "code": 901,
      "primitivas": ["MultiLinestring", "MultiPoint"]
    },
    {
      "code": 1001
    }
  ]
}

Outro modo de organizar as primitivas geométrica é da seguinte forma:

{
  "nome": "tipo",
  "tipo": "smallint",
  "cardinalidade": "1..1",
  "mapa_valor": "tipo_elemento_infraestrutura",
  "valores": {
    "MultiPoint": [ 901, 1001 ],
    "MultiLinestring": [ 701, 801, 1001 ],
    "MultiPolygon": [ 701, 801, 901, 1001 ]
  }
}

Certos atributos podem ser limitados a uma primitiva geométrica da seguinte forma:

{
  "nome": "simb_rot",
  "tipo": "real",
  "cardinalidade": "0..1",
  "primitivas": [ "MultiPoint" ]
}

Classes auxiliares​

Classes auxiliares segue o mesmo padrão de classes, só estando separado para fins de organização.

Modelagens​

As seguintes modelagens possuem Masterfile:

  • EDGV 2.1.3
  • EDGV 2.1.3 Shapefile
  • EDGV 2.1.3 Pro
  • EDGV 3.0
  • EDGV 3.0 Shapefile
  • EDGV 3.0 Pro
  • EDGV 3.0 Pro Orto
  • TRD 4.4
  • TRD 4.4 Shapefile

Existem masterfiles distintos para implementação em Shapefile pois tal implementação requer um padrão distinto de nome de classes e de atributos. Seria interessante no futuro estender a definição do Masterfile para comportar ambos os casos de forma única.

Todos os Masterfiles estão disponíveis no repositório de modelagens da DSG.

Rotinas​

As seguintes rotinas estão implementadas:

  • Cria SQL a partir do Masterfile
  • Cria Gpkg a partir do Masterfile
  • Cria Shapefile a partir do Masterfile

JSON Schema​

Para validar o Masterfile é necessário utilizar o JSON Schema que também está disponível no repositório de modelagens da DSG.
